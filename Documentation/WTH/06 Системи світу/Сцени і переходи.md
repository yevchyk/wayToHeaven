# Сцени і переходи

Сюжетні notes мають відображати не лише тему сцени, а й її функцію в переході. Для runtime важливо, щоб у сцени був один канонічний формат.

## Runtime контракт

- канонічний authored формат сцени це `scene-generation -> sceneFlow`
- `sceneFlow` є source of truth
- `dialogue`, `travel board`, `city hub` це runtime-представлення або projection
- нові сцени не варто authorити в обхід `sceneFlow`, якщо вони вже входять у цей пайплайн

## Базова схема

- сцена
- хто присутній
- де відбувається
- який конфлікт оголює
- який стан історії змінює
- куди веде далі

## Стиль authored сцени

- note сцени і runtime-текст мають бути конкретними й функціональними
- narration має фіксувати ситуацію, тиск, ризик, реакцію або наслідок, а не працювати як літературний монолог
- якщо beat створює атмосферу, вона все одно має бути прив’язана до ігрової загрози, соціального правила або зміни стану
- вступи на кшталт “це була не подія, а середовище” або інша декоративна абстракція не є канонічним стилем для authored content
- `WeyToHeaven` authorиться як dark adult visual novel з еротизованими і подекуди порнографічно зарядженими beat'ами там, де влада проходить через тіло; це не дитяча казка і не sanitized noble fantasy
- персонажі не повинні звучати як безкінечний генератор афоризмів; у діалогах потрібні конкретні речі світу: охорона, повстанці, гроші, мода, релігія, побут, логістика, борги, статус

## Authoring flow

1. Спочатку створюється або оновлюється note сцени в `Documentation` за [[Шаблон сцени]].
2. Потім сцена звіряється з atomic-notes персонажів, локацій і систем, а не з одним великим “загальним лором”.
3. Після цього зміст переноситься в authored `scene-generation` документ.
4. Уже готовий runtime-прохід перевіряється в Storybook як візуальний workbench, а не як місце, де вигадується канон.

## Флоу сценічних ефектів

Для runtime-позицій персонажів, підсвітки, масштабування і transition-перескладання сцени порядок такий:
1. Спершу authorиться executable `scene-generation` JSON/TS з `stage.characters[].placement`.
2. Потім runtime перевіряється в грі або workbench, поки placement і focus-переходи не працюють як треба.
3. Одразу після цього рішення фіксується в `Documentation`, щоб documented contract лишався джерелом істини, а не відставав від runtime.

Для screen transition є окреме правило:
- transition це presentation-trigger, а не декоративне блимання на кожному node advance
- його варто використовувати для реальної зміни сцени, background, CG, overlay або іншого помітного presentation patch
- якщо beat змінює лише текст або локальний вибір без візуального зсуву, transition там не потрібен

Для scene effects діє окреме правило:
- ефекти типу `blood border`, `shadow veil`, `smoke haze` не authorяться як окремі локації
- canonical місце для них це `backgroundStyle` на scene-level або `sceneChange.background.style` на beat-level
- якщо effect має жити від самого відкриття сцени, він іде в `meta.defaultBackgroundStyle` або `scene.backgroundStyle`
- якщо effect потрібен лише на одному зсуві всередині сцени, він іде в `sceneChange.background.style`

Для erotic / adult image-sequence beats діє окреме правило:
1. якщо сцена працює як спостереження, неправильна близькість або image-led shock beat, її не треба ставити як звичайний portrait dialogue
2. у таких сценах `stage` може бути очищений, а акцент переноситься на background / CG / image-state transitions
3. canonical note для цього шару: [[Еротизовані image-sequence сцени]]

## Mapping note -> runtime

- назва note стає `scene.title`, якщо це повна runtime-сцена
- якщо note описує лише один beat усередині більшої сцени, його назва стає `node.title`
- `Де` зазвичай впливає на `scene.locationName`, `scene.backgroundId` або `node.backgroundId`
- `Хто в сцені` впливає на `scene.stage.characters`, `node.speakerId` і портретні стани персонажів
- `Що відбувається` розкладається на `scene.description`, `node.text` і `choices[*].text`
- `Навіщо сцена` не переноситься дослівно; воно має стати умовами, ефектами, переходами, змінами стану й виборами
- `Куди веде далі` має відбитися в `nextNodeId`, `nextSceneId`, `openSceneFlowId` або відповідних `GameEffect`

## HTML-контракт для тексту

- `text`, `description` і тексти виборів можуть містити safe subset HTML
- дозволені теги: `br`, `strong`, `b`, `em`, `i`, `u`, `p`, `ul`, `ol`, `li`, `span`, `code`
- runtime зрізає небезпечні теги й не зберігає довільні HTML-атрибути
- це authoring-інструмент для акценту, ритму абзаців і коротких inline-виділень, а не для верстки сцени

Мінімальний приклад:

```json
{
  "text": "<p>Тиша стала <strong>занадто щільною</strong> для звичайної чемності.</p><p><em>Вона вже працює як погроза.</em></p>"
}
```

## Storybook роль

- `Location Backdrop/Workbench` тепер має `Location State Prompt Composer` для master location -> room -> scene effects -> state authoring
- той самий workbench також лишає runtime reference grids, щоб видно було, у яке поле content-файлу підключений конкретний `backgroundId`
- `Narrative Portrait/Workbench` перевіряє reusable portrait refs і explicit scene overrides
- Storybook не замінює runtime QA для branching, conditions, effects, inventory/state changes і route logic

## DEV console bridge

У dev runtime відкриває browser-console bridge через `window.wthRuntime`.

- `window.wthRuntime.current()` повертає поточний runtime snapshot
- `window.wthRuntime.currentRef()` повертає короткий active ref на кшталт `chapter-1/scene/intro`
- `window.wthRuntime.print()` ще раз друкує в консоль тільки цей короткий ref
- у snapshot входять `activeRuntimeLayer`, `activeScreen`, `sceneFlow.sceneId`, `sceneFlow.flowId`, `sceneFlow.nodeId`, `dialogue.speakerId`, `world.locationId`, `battle.battleId`
- runtime автоматично пише в консоль тільки active ref, без debug-об’єкта
- цей bridge є authoring/debug-інструментом і не повинен ставати джерелом gameplay-логіки

## Наступний великий перехід

Після `chapter-1/scene/awakening` канонічне продовження слід читати як вихід у [[Огляд арки Караван до Гуген-Уму]].

Важлива межа:

- підрічкове пробудження з [[Нер-Азет]] лишається окремим переходом
- караванне пробудження у возі є вже входом у наступну соціально-виживальну фазу, а не заміною метафізичного aftermath-шару

## Наслідок для розробки

- адаптери з `sceneFlow` у `dialogue` чи `travel board` це нормально
- дублювати authored truth у кількох registry вже не варто
- якщо десь є legacy view-формат, він має трактуватися як похідний шар, а не як друга істина
