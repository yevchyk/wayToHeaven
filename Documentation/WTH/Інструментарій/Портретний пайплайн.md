# Портретний пайплайн

## Призначення

Цей документ фіксує затверджену архітектуру портретів для `WayToHeaven`, щоб runtime, Storybook і production-арт не розходилися між собою.

## Головне правило

У проєкті є дві різні подачі персонажів:

- `flat portrait`
- `heroine composite`

Вони не рівнозначні.

## Затверджений контракт

За замовчуванням усі персонажі працюють через `flat portrait`.

Це означає:

- кожна емоція є окремим цілісним зображенням
- scene data звертається до емоції через `portraitRefs` або до одного виняткового кадру через explicit `portraitId`
- для NPC ми не збираємо персонажа з `body + head + clothes`

`heroine composite` лишається окремим привілейованим пайплайном для головної героїні.

## Хто зараз є головною героїнею для runtime

Поточний story-facing id головної героїні: `mirella`.

Саме вона є основною героїнею в authored scenes і саме під неї треба приймати нові рішення щодо composite pipeline.

Legacy id `heroine` може лишатися в репозиторії для старих starter slices або тестів, але він не повинен задавати новий production-напрям для арту.

## Правило вибору в runtime

Runtime читає `portraitPresentation` у `NarrativeCharacterData`.

Поведінка така:

- якщо поле не задане, персонаж показується як `flat`
- якщо задано `composite`, runtime намагається показати layered heroine render
- explicit `portraitId` і outfit-specific portrait art мають право перекривати composite
- якщо composite-асети ще відсутні або залишаються placeholder-only, runtime автоматично повертається до flat portrait

## Практичний арт-пайплайн

### Головна героїня

Для героїні можна використовувати:

- `body/base`
- `clothes/base`
- `head/<emotion>`
- за потреби `hair/base`
- `hands/left`
- `hands/right`
- `weapon/base`

Тут має сенс працювати з повним staged rig, бо героїня:

- найчастіше з'являється в кадрі
- найчастіше змінює стани
- найбільше виграє від pose-based staging

### NPC і supporting cast

Для них базовий шлях інший:

- окремий цілісний портрет на кожну потрібну емоцію
- без modular head/body pipeline
- без масового розмноження composite rigs

Це дешевше по арту, простіше для авторингу і краще масштабується під великий каст.

## Storybook контракт

Для цього пайплайну Storybook теж розділяється:

- `Character Composite/Workbench` — тільки для головної героїні
- `Narrative Portrait/Workbench` — основний workbench для NPC flat emotion sheets

У Narrative Portrait workbench героїня може лишатися як:

- fallback portraits
- archive continuity
- one-off event art

Але це не повинно знову перетворити всіх персонажів на composite-систему.

## Коли можна відкривати composite для іншого персонажа

Тільки якщо одночасно виконуються всі умови:

- персонаж recurring і справді центральний для багатьох сцен
- flat emotion sheets уже перестають покривати staging-потребу
- є підтвердження, що це не роздує production cost
- runtime і docs свідомо оновлюються під новий виняток

Поки що таких винятків не затверджено.

## Що вважається помилкою

- тихо додати `images/characters/<npc-id>/...` і вважати це новою нормою
- почати робити `body + head` для кожного side character
- замінювати `portraitRefs` layered-рігом там, де достатньо звичайної емоції
- приймати арт-рішення для `mirella` через legacy `heroine` id
